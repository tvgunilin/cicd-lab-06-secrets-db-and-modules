Solo · Part 1 · Secrets
Part 1 · solo · ±30 min · exercises/lab.md
Hook up a secret for the db-connection

The warm-up left both database connections Faulted on develop. Fix them the way our production setup runs: passwords as secret files, a file-based secret provider resolving them, and a full deploy that ships the fix.

  • 1A: both database passwords become secret files at /run/secrets/: two connections, two users, two files.
  • 1B: a file-based secret provider on the gateway resolves them, and both connections point at referenced secrets.
  • 1C: build the fix for develop: a dev deployment-mode override plus the deploy workflow writing the secret files from GitHub secrets.
  • 1D: ship it as a full deploy moment: feature branch → PR → validation → merge → pipeline → develop's connections Valid.
Mustry Academy · CI/CD for IgnitionPart 1 · A → B → C → D
Part 1 · A
1A
Turn the Postgres passwords into secret files: one per database user
Two connections, same database server, different users: TimescaleDB logs in as ignition, TimescaleDB_Reports as the read-only reporting user. Each password goes from an env var anyone can docker inspect to a read-only file.
1
See the leak first: docker inspect lab06-timescaledb | grep PASSWORD → both values in plaintext.
2
Create both files from the committed examples: secrets/postgres_password.txt and secrets/reporting_password.txt. Keep the same values: Postgres only sets passwords on first volume init.
3
Edit docker-compose.yaml: top-level secrets: block with both files, POSTGRES_PASSWORD_FILE on the DB service, and both secrets attached to the gateway service: its provider reads them in 1B.
4
Recreate and prove it: inspect shows no passwords; both files sit at /run/secrets/ inside the containers.
# docker-compose.yaml
secrets:
  postgres_password:
    file: ./secrets/postgres_password.txt
  reporting_password:
    file: ./secrets/reporting_password.txt

services:
  timescaledb:
    secrets: [postgres_password]
    environment:
      POSTGRES_PASSWORD_FILE: /run/secrets/postgres_password
  gateway-loc:
    secrets: [postgres_password, reporting_password]

# step 4: the proof
docker inspect lab06-timescaledb | grep PASSWORD
docker exec lab06-gateway-loc ls -l /run/secrets/
Same value

Changing a password here won't change it in an existing database volume. This step changes where the value lives, not the value.

1B
Create the file-based secret provider and reference both passwords
After this step the gateway config holds names, not values: that's what makes it safe to commit and able to travel to develop and prod.
1
On the local gateway, create a file-type secret provider named LabSecrets.
2
Add two secrets to it: POSTGRES_PASSWORD → /run/secrets/postgres_password and REPORTING_PASSWORD → /run/secrets/reporting_password.
3
Re-point both connections' passwords at their referenced secret: TimescaleDB → POSTGRES_PASSWORD, TimescaleDB_Reports → REPORTING_PASSWORD. Test: both stay Valid.
4
Prove nothing leaked: grep the exported config for both password values → no hits. All that's committed is the "type": "Referenced" shape from the teaching.
# what lands in the committed config (per connection)
"password": {
  "type": "Referenced",
  "data": {
    "providerName": "LabSecrets",
    "secretName": "REPORTING_PASSWORD"
  }
}

# step 4: prove the export is clean
grep -r "$(cat secrets/postgres_password.txt)" \
  services/config/ || echo "clean ✅"
grep -r "$(cat secrets/reporting_password.txt)" \
  services/config/ || echo "clean ✅"
Names travel

The provider and the references are ordinary config: they deploy. The files behind them don't, and mustn't. 1C is about getting those files onto the develop host.

Mustry Academy · CI/CD for IgnitionPart 1 · B of D
Part 1 · C
1C
Build the fix for develop: dev deployment-mode override plus secret files from CI
Develop faults for two reasons, so the fix has two halves: point the reports connection at the right database with a dev deployment-mode override (the Lab 04 mechanism), and let the deploy workflow write the secret files.
1
Diagnose: TimescaleDB_Reports only exists in core, so develop targets ignition_loc. And neither secret file exists on the develop host yet.
2
Fix the target: add the dev override …resources/dev/ignition/database-connection/TimescaleDB_Reports/config.json with connectURL → ignition_dev. Copy the pattern from TimescaleDB, which already has one.
3
Fix the values: in your fork, add POSTGRES_PASSWORD and REPORTING_PASSWORD as secrets on the lab-gateway-dev environment.
4
Add the Materialize secret files step to deploy.yml, before compose up: umask 177 + printf, one file per secret.
5
Sanity check locally: scripts/validate.sh green, secret scan zero findings, git status shows no secrets/*.txt.
# deploy.yml: the step you add
- name: Materialize secret files
  env:
    POSTGRES_PASSWORD: ${{ secrets.POSTGRES_PASSWORD }}
    REPORTING_PASSWORD: ${{ secrets.REPORTING_PASSWORD }}
  run: |
    umask 177
    mkdir -p secrets
    printf '%s' "$POSTGRES_PASSWORD" \
      > secrets/postgres_password.txt
    printf '%s' "$REPORTING_PASSWORD" \
      > secrets/reporting_password.txt
Nothing deployed yet

Everything so far is local edits plus GitHub settings. Develop is still Faulted, on purpose: the fix only lands through the pipeline, and that's 1D.

Ship a schema change as a migration the pipeline applies

db-init/ is bootstrap: it never touches an existing database. A schema change ships as a numbered golang-migrate pair, applied by deploy.yml before the screens that need it.

  • 2A: write 0002_add_downtime_log as an up/down pair, apply it locally, read schema_migrations.
  • 2B: add the migrate step to deploy.yml before the ship step, and watch the pipeline migrate dev.
Mustry Academy · CI/CD for IgnitionPart 2 · A → B
Part 2 · A
2A
Write migration 0002 as an up/down pair, apply it locally
Golang-migrate conventions, exactly like our file-based production repo: 4-digit number, descriptive name, always a pair.
1
Create db-migration/migrate/0002_add_downtime_log.up.sql and …0002_add_downtime_log.down.sql. The down undoes exactly what the up does.
2
Apply it locally: scripts/migrate.sh up (golang-migrate in Docker, like the production migrate.sh).
3
Read the ledger: the schema_migrations table now says version 2, not dirty. Run migrate.sh up again → no change: it only applies what's missing.
4
Now the rule the tool won't enforce for you: edit an already-applied migration file. Nothing complains, and that's the trap: other environments would silently diverge. Undo the edit; changes go in 0003. That's why the production repo's docs/MIGRATIONS.md makes it a written rule.
# 0002_add_downtime_log.up.sql
CREATE TABLE downtime_log (
  id BIGSERIAL PRIMARY KEY,
  reason TEXT NOT NULL,
  minutes INT NOT NULL,
  area TEXT
);

# 0002_add_downtime_log.down.sql
DROP TABLE downtime_log;

# apply + inspect the ledger
scripts/migrate.sh up
docker exec lab06-timescaledb psql -U ignition \
  -d ignition_loc -c 'SELECT * FROM schema_migrations;'
Pairs, always

"All migrations must come in pairs" and "never edit a deployed migration" are the two production rules that keep every environment replayable from zero. The tool tracks position; the discipline is yours.

Mustry Academy · CI/CD for IgnitionPart 2 · A of B
Part 2 · B
2B
Add the migrate step to deploy.yml, before the ship step
Old screens tolerate a new table; new screens don't tolerate a missing one. So: migrate first, and a migrate failure stops the whole run.
1
Add the Migrate database step to .github/workflows/deploy.yml, before the ship-files step, targeting the dev database. The DB password comes from the secret file Part 1 set up.
2
Branch, add a small screen change that reads downtime_log, and open a PR with both the migration and the screen: schema and screens travel in the same PR.
3
Merge and watch the run: validate → migrate dev DB → ship files → scan → verify.
4
Prove it was the pipeline, not you: query dev's schema_migrations: version 2, applied during the run.
# deploy.yml: the step you add
- name: Migrate database
  run: scripts/migrate.sh up --database ignition_dev
# placed BEFORE the ship/scan steps, and NOT
# continueOnError: a failed migration stops the run.

# step 4: the proof, against the dev database
docker exec lab06-timescaledb psql -U ignition \
  -d ignition_dev -c 'SELECT * FROM schema_migrations;'
✅ Part 2 gate

A green deploy run whose log shows migrate before ship, and dev's ledger at version 2. Bonus debrief point: the production pipeline you saw runs this step after the scan with continueOnError: true. What can go wrong there that can't go wrong in yours?

Optional stretch
OPTIONAL
Stretch: three ways to go deeper
Pick by appetite. Each one extends a different part of the hour.
1
The internal secret provider, and where it breaks. On your local gateway, create an internal secret provider and store REPORTING_PASSWORD in it: the gateway encrypts the value and keeps the ciphertext in its own config. Point TimescaleDB_Reports at it: locally it stays Valid. Now ship that config through the pipeline and check develop: Faulted, because the ciphertext only decrypts on the gateway that created it. Explore ignition-secrets-tool.sh (shared root key + KEK under data/config/ignition/keys/) as the escape hatch, then revert to the referenced secret and write down which trade-off you'd pick.
2
Expand-contract rename. Rename a column with two migrations (0003 add + backfill, 0004 drop) and a screen change in between, so no deploy ever breaks a live query.
3
Secret scanner in CI. Add a gitleaks job to ci.yml (with fetch-depth: 0: the scanner must see history, not the tip). Test it with a fake-key PR → red ❌ → close without merging.
# stretch 3: ci.yml secret-scan job
secret-scan:
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4
      with:
        fetch-depth: 0
    - uses: gitleaks/gitleaks-action@v2
Two philosophies

Stretch 1 and Part 1 are competing answers to the same problem: references + files, or gateway-encrypted ciphertexts + shared keys. Both are defensible; know what each makes easy and what each makes dangerous.
